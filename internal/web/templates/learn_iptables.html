{{template "icon-compass"}}The mental shift

The rules look similar; the structure doesn't. Three things change how you think:

Nothing is built in

iptables hands you filter/nat tables and INPUT/FORWARD/OUTPUT chains. nftables starts empty — you create the tables and chains you want, and a base chain only exists because you hooked it.

Many chains per hook

iptables has one built-in chain per hook. nftables lets several base chains attach to the same hook, ordered by priority — so you can keep concerns in separate chains.

Verdict, not target

There's no -j. A rule ends in a verdict (accept, drop, jump X) written inline, and a rule can carry several statements before it.

{{template "icon-shield"}}One ruleset for v4 and v6

The big day-to-day win: no more maintaining iptables and ip6tables in parallel. A table in the inet family sees both — write the rule once. Where you do need a family-specific match (ip saddr vs ip6 saddr), you add one of each, and named sets render a @name4 and @name6 for exactly that. (If mixed results ever surprise you, the family mismatch note explains why.)

{{template "icon-ban"}}Sets & maps are built in

Everything you used ipset for is native. A named set holds addresses or ranges and a rule points at it with @name; edit the set in one place and every rule follows. Sets can be filled from a country (GeoIP) or a remote feed, and a dynamic set with a timeout is how the auto-ban works — the kernel adds and expires members itself, no ipset commands, no cron.

{{template "icon-changes"}}Side by side

A default-deny host that allows established traffic and SSH from one network:

iptables

iptables -P INPUT DROP
iptables -A INPUT -i lo -j ACCEPT
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
iptables -A INPUT -s 203.0.113.0/24 -p tcp --dport 22 -j ACCEPT

nftables (what nftably builds)

table inet filter {
	chain input {
		type filter hook input priority filter; policy drop;
		iifname "lo" accept
		ct state established, related accept
		ip saddr 203.0.113.0/24 tcp dport 22 accept
	}
}

Same logic, one block, and it already covers IPv6 for everything except the v4-specific ip saddr line (add an ip6 saddr twin for v6 SSH).

{{template "icon-import"}}Bring your rules across

nftably can show your current iptables ruleset translated to nftables syntax, as a reference to rebuild from.

  • Open Settings → Import — it runs iptables-restore-translate over what's on the box and shows the nftables equivalent. Nothing is applied; it's a preview.
  • Use it to rebuild the rules in the Firewall editor, where each goes out under the armed auto-revert — a safer path than importing wholesale.
