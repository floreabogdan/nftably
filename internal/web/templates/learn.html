{{template "icon-route"}}The packet's journey

Every packet the kernel handles passes through a fixed set of hooks — checkpoints on its way in, out, or through the box. A firewall is just rules attached to those checkpoints. Which hook a packet hits depends on where it's going:

arrivesfrom the wire
→
preroutingbefore routing decision
→
inputfor this box
forwardpassing through (router)
→
outputfrom local apps
→
postroutinglast stop out
→
leavesto the wire
input

Traffic addressed to this machine — someone connecting to your SSH, web server, or this UI. This is the hook you'll spend most of your time on when hardening a single host.

forward

Traffic routed through this box to somewhere else. Only matters if the machine is a router or gateway. On a plain server it's empty.

output

Traffic this machine originates. Usually left open, but you can restrict what your box is allowed to reach — handy for locking down a server that should only talk to a few places.

prerouting / postrouting

The very first and very last steps. NAT lives here (rewriting addresses), as does early filtering. Most host firewalls don't need them.

{{template "icon-list"}}Tables & families

A table is a container that holds your chains and rules. Every table belongs to a family, which decides what kind of traffic it can see:

inet recommended

IPv4 and IPv6 together. One set of rules covers both — the modern default, and what nftably's presets use. Write a rule once instead of maintaining two parallel firewalls.

ip / ip6

IPv4-only and IPv6-only. Use these only when a rule genuinely applies to one protocol and not the other.

arp / bridge / netdev

Specialist families for link-layer and bridged traffic, or ultra-early filtering at the driver. Advanced; you'll know when you need them.

Tables and chains are just names and structure — nothing filters until you add rules to a base chain (next).

{{template "icon-shield"}}Chains: base vs regular

A chain is an ordered list of rules. There are two kinds, and the difference is the single most common source of confusion:

Base chain attached to a hook

Hooked into the packet journey above, so the kernel actually runs it. A base chain declares four things:

  • hook — which checkpoint (input, forward, output…).
  • type — filter (allow/deny), nat, or route.
  • priority — the order among chains on the same hook; lower numbers run first. This is how your filtering runs before or after NAT.
  • policy — the verdict if no rule matches: accept (permit by default) or drop (deny by default).
Regular chain not hooked

Not attached to any hook, so it never runs on its own. It's a reusable block of rules you jump to from a base chain — like a function. Great for grouping (e.g. a "web-services" chain you jump to for ports 80/443), keeping big rulesets readable.

{{template "icon-alert"}}
The golden rule of a secure host: set your input base chain's policy to drop, then explicitly accept only what you need. That's "default-deny" — the firewall permits nothing you didn't ask for. The Posture grades exactly this.
{{template "icon-changes"}}Rules: matches & verdicts

Each rule reads left to right: zero or more matches (conditions the packet must meet), then statements — usually ending in a verdict that decides the packet's fate. If every match is true, the statements run. nftably's editor builds exactly this, one dropdown at a time.

tcp dport 22 ip saddr 10.0.0.0/8 accept  // allow SSH, but only from the 10.x network

Common verdicts:

accept Let the packet through — stop evaluating this chain.
drop Discard silently. The sender gets nothing back (looks "dark" to a scanner).
reject Refuse politely, sending back an error. Faster failures for legitimate clients, but confirms the box exists.
jump / goto Divert to a regular chain (jump returns; goto doesn't).
continue No verdict — keep evaluating. A bare counter or log does this.
{{template "icon-shield"}}
Order matters: rules run top to bottom and the first verdict wins. On the Firewall page you drag rules to reorder them, and every change previews as real nft syntax before anything is applied.
{{template "icon-activity"}}Connection tracking (the "why nothing works" fix)

This is the idea that trips up every newcomer. Under a default-deny policy, if you only "allow SSH in", the reply packets from your SSH server heading back out — and the return traffic of every connection you start — get dropped, because no rule allows them. Everything breaks.

The kernel's connection tracker (conntrack) remembers every flow it has seen and labels each packet with a state. You accept traffic based on that label instead of writing rules for both directions:

new

The first packet of a connection nobody's seen yet. This is what your "allow SSH" rule is really matching.

established

Part of a connection already accepted. Accept these and every reply flows freely — one rule, both directions.

related

A new connection expected by an existing one (e.g. FTP data, ICMP errors). Usually accepted alongside established.

invalid

Packets conntrack can't place in any flow — malformed, or out of sequence. A favourite of scanners. Drop these early.

{{template "icon-info"}}
The survivable base of almost every host firewall is just two lines near the top: ct state established,related accept and ct state invalid drop. The Posture verifies you have both, and can add them in one click.
{{template "icon-ban"}}Sets & maps

A set is a named collection of values — addresses, ports, countries — that a rule can test against in one shot. Instead of a hundred rules for a hundred bad IPs, you write ip saddr @blocklist drop and manage the membership separately.

Named sets

Reusable lists you maintain in one place. nftably's Named sets page manages these — an allow-list of your networks, a block-list you can add to with one click from the Connections view, or a feed pulled from a URL.

Maps

Like a set, but each value points to a result — e.g. map a destination port to which chain handles it. Powerful for large, table-driven rulesets.

Dynamic sets

Sets rules can add to at runtime, often with a timeout. This is how native rate-limiting and auto-ban work: an address that trips a limit gets added to a "penalty" set for 10 minutes, no external daemon needed.

{{template "icon-shield"}}How nftably keeps you safe

Editing a live firewall is how people lock themselves out of a remote box. nftably is built so that can't happen by accident:

  • Model first, apply later. Everything you build on the Firewall page is a model — nothing touches the kernel until you go to Review.
  • See the diff. Before applying, you get the exact nft commands and a diff against what's live. No surprises.
  • Armed auto-revert. When you apply, nftably arms a timer and loads the new ruleset. If you don't confirm within the window — because you just locked yourself out — it automatically rolls back to the previous ruleset. Get it wrong and you're back in a minute.
