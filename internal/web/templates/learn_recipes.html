{{template "icon-shield"}}Open a service to everyone

Publish a web server (or any service) to the whole internet.

  • On the Firewall page, open your input chain and Add rule.
  • Condition: Destination port (TCP) = 80, 443. Action: Accept.
  • Save, then Changes to apply.
{{template "icon-route"}}
Confirm it first: simulate a packet to tcp/443 — it should read ACCEPT.
{{template "icon-ban"}}Allow SSH from just your network

SSH open to the world is the internet's most-scanned port. Scope it to a management set instead.

  • Under Named sets, make a set named mgmt and add your admin network (e.g. 203.0.113.0/24). It renders as @mgmt4 / @mgmt6.
  • On the Firewall page add an input rule: Source address (IPv4) = @mgmt4, Destination port (TCP) = 22, action Accept.
  • Don't add an open tcp dport 22 accept anywhere else, and let the chain's drop policy handle the rest.
{{template "icon-info"}}
Every preset already does this — apply one and just fill in @mgmt. The Security Posture page flags SSH that's open to the world.
{{template "icon-route"}}Port-forward to an inside host

Reach a server behind the router — the classic two-part port-forward.

  • In a prerouting nat chain, add a rule: Destination port (TCP) = 443, action Destination NAT (port forward) to 192.168.1.10.
  • In the forward chain, accept the rewritten flow: condition Connection status = dnat, action Accept.
  • Save, Changes, apply. The home router preset ships the empty prerouting chain ready for this.
{{template "icon-book"}}
New to this? NAT & port-forwarding explains why it takes two rules.
{{template "icon-activity"}}Rate-limit a public service

Soften a flood: let new connections through only up to a rate, and drop the excess.

  • Add an input rule that accepts the service but carries a Rate limit action — e.g. new connections to your port, Rate 25 per second, Burst 50.
  • Below it, a plain drop for the same port catches whatever exceeds the rate.
{{template "icon-shield"}}
For SSH specifically, don't hand-roll it — the auto-ban recipe below is stronger.
{{template "icon-shield"}}Auto-ban brute-force on any service

Fail2ban in the kernel — a source that hammers a service faster than a threshold is dropped into a timeout set and blocked, no daemon and no log parsing.

  • Go to Security Posture → Auto-ban any service. Click SSH or HTTP to prefill the form, or name your own service and give it a protocol and port (e.g. smtp / tcp / 25, 587, or a game server on udp).
  • It inserts, for IPv4 and IPv6: a detect-and-ban rule (over the rate you set → add the source to a timeout set) and an early drop of that set, above your allow rules.
  • Tune the rate, burst or ban time on the Firewall page — it's the Rate-ban the source action on the detect rule.
{{template "icon-info"}}
It's a dynamic timeout set the kernel fills and clears itself — offenders age out automatically. Unban someone early from Security Posture or the block API.
{{template "icon-ban"}}Block a country or a bad-IP feed

Drop traffic from a whole country's address space, or from a curated blocklist, kept refreshed.

  • Fastest: on the Connections view, click a remote address's country to build and drop that country's set in one click.
  • Or under Named sets, make a set sourced from a country (GeoIP) or a remote feed (URL), tick keep it refreshed, then add an early input rule: Source address = @yourset4, action Drop.
{{template "icon-search"}}
Rule not biting? Troubleshooting covers set references and rule order.
{{template "icon-spark"}}Speed up a router with a flowtable

On a box that forwards a lot of traffic, a flowtable hands established connections to the kernel's fast path — they skip the rest of the ruleset, for a big throughput win.

  • On the Firewall page, add a flowtable to your forward table and bind it to the interfaces traffic crosses — e.g. wan and lan. Turn on hardware offload only if your NIC supports it.
  • In the forward chain add a rule: condition Connection state = established, related, action Offload to flowtable (flow add @yourft), placed early.
  • Only the first packet or two of a connection still walks the rules; the rest is offloaded. Your security rules keep applying to new connections.
{{template "icon-alert"}}
The bound interfaces must exist when you apply — nftables rejects a flowtable naming a device that isn't there (unlike an iifname match, which is fine to pre-stage). This is why presets describe the flowtable instead of seeding one.
{{template "icon-activity"}}Account for traffic with named counters

A named counter is a running packet/byte total you can share across rules — "how much traffic did all my block rules drop?" in one number.

  • On any rule, add the Count action and give it a name (e.g. denied). Add the same-named counter to other rules and they all add into the one total.
  • nftably declares the counter denied { } object for you automatically — it's derived from the name you used, so there's no separate thing to create or delete.
  • Leave the name blank for a plain per-rule (anonymous) counter when you only care about that one rule.
{{template "icon-info"}}
The hardened presets already route denied inbound into a denied counter — apply one, then watch the number climb as scanners hit you.
{{template "icon-compass"}}Route many verdicts with one map

A verdict map (vmap) turns a long if/else ladder into a single lookup — map a key like the destination port straight to accept, drop or a jump.

  • Add a rule and choose the Verdict map action. Pick the key to look up (e.g. TCP destination port), then list the entries: 22 : accept, 80 : accept, 443 : accept, and so on.
  • One rule now replaces one-per-port — faster to match and far easier to read than a stack of accepts.
  • Entries can jump to a chain (25 : jump mail) to branch into a whole sub-policy per key.
{{template "icon-route"}}
Prove it before applying: simulate a packet on one of the mapped ports and confirm it lands on the verdict you expect.
{{template "icon-edit"}}Write a rule nftably can't express yet

The catalogue covers the common cases; the raw rule is the escape hatch for the rest — you type a verbatim nft line and nftably places it for you.

  • On the Firewall page, add a rule and switch it to Raw. Type one nft rule line, e.g. ip saddr 10.0.0.0/8 ct count over 20 drop.
  • nftably validates the shape (single line, balanced braces, no rule-separators or comments — nothing that could break out of the chain) and every apply still passes the atomic nft --check first.
  • Tag it so future-you remembers why it's hand-written — tags show on the rule and group related ones together.
{{template "icon-alert"}}
Referencing a named counter from a raw line? Declare that counter by adding a normal Count action with the same name on any rule — a raw line alone won't auto-declare it, and apply would reject an undeclared counter.
