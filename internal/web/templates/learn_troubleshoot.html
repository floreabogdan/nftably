{{template "icon-list"}}Order, and how a chain is evaluated

Rules in a chain are read top to bottom. When a rule matches and carries a verdict (accept, drop, reject), evaluation stops there — that's the decision. A rule that matches but has no verdict (just a counter or log) does its thing and evaluation continues to the next rule.

So the classic bug: a broad accept above your specific rule. If tcp dport 22 accept sits above your ip saddr @banned drop, the accept already decided — the drop never runs. Put the narrow, protective rules (drops, bans) above the broad accepts.

{{template "icon-activity"}}The established-accept short-circuit

Nearly every firewall starts with ct state established, related accept — it lets the replies to allowed traffic back in. But it has a side effect that confuses people: a connection that was already allowed keeps flowing even after you add a rule to block it.

You add ip saddr 1.2.3.4 drop, but 1.2.3.4's existing SSH session stays up — because its packets match established first and get accepted before they ever reach your new drop. New connections are blocked; the live one isn't. To cut even active sessions, the drop has to sit above the established-accept — which is exactly why nftably's Block button and the presets put the blacklist drop first.

{{template "icon-route"}}new vs established

A rule matched on ct state new only sees the first packet of a connection. If your rule doesn't match when you expect, check whether you've pinned it to a state that doesn't apply.

new

The opening packet. Right for rules that decide whether to allow a connection — accepts, rate-limits, auto-bans.

established, related

Packets belonging to a connection already allowed (and its children, like FTP data). The "let replies back" rule.

invalid

Packets conntrack can't place in any connection — drop these early; matching anything else on them rarely does what you want.

{{template "icon-shield"}}Address-family mismatch

In the inet family a single rule sees both IPv4 and IPv6 packets — but an ip saddr … match only applies to IPv4, and ip6 saddr … only to IPv6.

So a rule written only with ip saddr @office silently ignores all your IPv6 traffic — it's not matching "nothing", it's matching no v6. When you scope by address, cover both families (that's why named sets render as @name4 and @name6, and presets add a rule for each). If a rule seems to work for some clients and not others, family is the first suspect.

{{template "icon-list"}}A chain that never runs

Only base chains — the ones with a hook (input, forward, output…) — are run by the kernel automatically. A regular chain is just a named list; it does nothing until another rule sends traffic to it with a jump or goto.

So if you grouped rules into a regular chain and "none of them fire", check that a base chain actually jumps to it. And a rule that won't render at all shows a won't render badge on the Firewall page — usually an empty value or a knob left half-filled; open it and complete the field.

{{template "icon-search"}}Stop guessing — see it

nftably is built so you never have to apply-and-pray:

  • The packet simulator traces a packet you describe through your model, hook by hook, and names the exact rule that decides it — the fastest way to answer "why did this get dropped?".
  • Live counters. A rule with a Count action shows its running packet/byte total on the Firewall page — zero counts on a rule you expect to fire means traffic isn't reaching it.
  • The firewall log. Add a Log action above a drop to see exactly what's being blocked, live from the kernel.
  • The Live ruleset shows what's actually loaded — compare it against your model on the Changes page's diff.
