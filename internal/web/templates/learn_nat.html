{{template "icon-route"}}What NAT actually is

Network Address Translation is the kernel rewriting the source or destination address (and often the port) of a packet as it passes through. Two everyday jobs:

Share one connection

Your LAN uses private addresses (192.168.0.0/16, 10.0.0.0/8) that can't appear on the internet. The router rewrites their source to its own public IP on the way out, and puts the replies back — source NAT.

Publish an inside service

A server on your LAN has no public address. The router rewrites the destination of traffic arriving on a chosen port to that inside host — a port-forward, or destination NAT.

NAT is not a firewall. It rewrites addresses; it doesn't decide what's allowed — that's the filter chains' job (Concepts covers those). The two work together, which is exactly where people get confused, so we'll come back to it.

prerouting

Runs before the kernel decides where a packet goes — so this is where you rewrite the destination (a port-forward), so routing then sends it to the right inside host. Priority dstnat.

postrouting

Runs after routing, just before the packet leaves — the last chance to rewrite the source so replies come back. Masquerade lives here. Priority srcnat.

{{template "icon-shield"}}Masquerade vs SNAT

Both rewrite the source address on the way out. The difference is whether your public address is fixed:

masquerade

Uses whatever address the outgoing interface currently has — perfect for a home connection whose IP changes. Slightly more work per packet, but you never have to name the address. oifname "wan" masquerade.

snat to <addr>

Rewrites to a fixed address you specify — use it when you have a static public IP; it's a touch cheaper. snat to 203.0.113.10.

When in doubt, masquerade — it always works. Put it in a postrouting nat chain, matched on your uplink interface so only outbound traffic is translated.

{{template "icon-route"}}Port-forwarding, step by step

Say a web server lives at 192.168.1.10 on your LAN and you want the internet to reach it on port 443. Two pieces have to line up:

  • The rewrite (DNAT). In a prerouting nat chain, match the traffic and rewrite its destination: tcp dport 443 dnat to 192.168.1.10. Now routing sends it inside.
  • The permission (filter). DNAT only changes the address — the packet still has to survive your forward filter chain. Under a default-drop forward policy you need a rule that accepts it: ct status dnat accept lets through exactly the flows a port-forward rewrote, nothing more.

{{template "icon-info"}}NAT rewrites, filter decides

The single most common NAT confusion: a port-forward that doesn't work because the filter chain drops it. Remember the order — a packet hits the nat prerouting chain (address rewritten), then routing, then the forward filter chain (allowed or not), then nat postrouting.

So a working port-forward is always two rules: the DNAT, and a forward-chain accept. If a forward you're sure about isn't landing, the filter chain is the first place to look — trace it in the simulator, which shows every hook the packet passes and the exact rule that decides it.
